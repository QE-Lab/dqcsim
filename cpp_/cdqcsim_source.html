279 /// Type for a handle.
280 ///
281 /// Handles are like pointers into DQCsim's internal structures: all API calls
282 /// use these to refer to objects. Besides the object, they contain type
283 /// information. This type can be retrieved using `dqcs_handle_type()`.
284 ///
285 /// Handles are always positive integers, counting upwards from 1 upon
286 /// allocation, and they are not reused even after being deleted. Thus, every
287 /// subsequent object allocation returns a handle one greater than the
288 /// previous. Note however that DQCsim may allocate objects as well without
289 /// the user specifically requesting this, so external code should generally
290 /// *not* rely on this behavior unless otherwise noted. The value zero is
291 /// reserved for invalid references or error propagation.
292 ///
293 /// Note that the scope for handles is thread-local. That is, data referenced
294 /// by a handle cannot be shared or moved between threads.
295 ///
296 /// The value zero is reserved for invalid references or error propagation.
297 using dqcs_handle_t = unsigned long long;
298 
299 /// Type for a qubit reference.
300 ///
301 /// Qubit references are exchanged between the frontend, operator, and backend
302 /// plugins to indicate which qubits a gate operates on. Note that this makes
303 /// them fundamentally different from handles, which are thread-local.
304 ///
305 /// Qubit references are always positive integers, counting upwards from 1 upon
306 /// allocation, and they are not reused even after the qubit is deallocated.
307 /// Thus, every subsequent allocation returns a qubit reference one greater
308 /// than the previous. This is guaranteed behavior that external code can rely
309 /// upon. The value zero is reserved for invalid references or error
310 /// propagation.
311 using dqcs_qubit_t = unsigned long long;
312 
313 /// Type for a plugin state.
