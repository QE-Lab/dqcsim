101 /// Types of DQCsim gates.
102 enum class dqcs_gate_type_t {
103  /// Invalid gate type. Used as an error return value.
104  DQCS_GATE_TYPE_INVALID = 0,
105  /// Unitary gates have one or more target qubits, zero or more control
106  /// qubits, and a unitary matrix, sized for the number of target qubits.
107  ///
108  /// The semantics are that the unitary matrix expanded by the number of
109  /// control qubits is applied to the qubits.
110  ///
111  /// The data field may add pragma-like hints to the gate, for instance to
112  /// represent the line number in the source file that generated the gate,
113  /// error modelling information, and so on. This data may be silently
114  /// ignored.
115  DQCS_GATE_TYPE_UNITARY,
116  /// Measurement gates have one or more measured qubits and a 2x2 unitary
117  /// matrix representing the basis.
118  ///
119  /// The semantics are:
120  ///
121  /// - the hermetian of the matrix is applied to each individual qubit;
122  /// - each individual qubit is measured in the Z basis;
123  /// - the matrix is applied to each individual qubit;
124  /// - the results of the measurement are propagated upstream.
125  ///
126  /// This allows any measurement basis to be used.
127  ///
128  /// The data field may add pragma-like hints to the gate, for instance to
129  /// represent the line number in the source file that generated the gate,
130  /// error modelling information, and so on. This data may be silently
131  /// ignored.
132  DQCS_GATE_TYPE_MEASUREMENT,
133  /// Prep gates have one or more target qubits and a 2x2 unitary matrix
134  /// representing the basis.
135  ///
136  /// The semantics are:
137  ///
138  /// - each qubit is initialized to |0>;
139  /// - the matrix is applied to each individual qubit.
140  ///
141  /// This allows any initial state to be used.
142  ///
143  /// The data field may add pragma-like hints to the gate, for instance to
144  /// represent the line number in the source file that generated the gate,
145  /// error modelling information, and so on. This data may be silently
146  /// ignored.
147  DQCS_GATE_TYPE_PREP,
148  /// Custom gates perform a user-defined mixed quantum-classical operation,
149  /// identified by a name. They can have zero or more target, control, and
150  /// measured qubits, of which only the target and control sets must be
151  /// mutually exclusive. They also have an optional matrix of arbitrary
152  /// size.
153  ///
154  /// The semantics are:
155  ///
156  /// - if the name is not recognized, an error is reported;
157  /// - a user-defined operation is performed based on the name, qubits,
158  /// matrix, and data arguments;
159  /// - exactly one measurement result is reported upstream for exactly the
160  /// qubits in the measures set.
161  DQCS_GATE_TYPE_CUSTOM,
162 };

306  /// reported to the API/function caller via Result::Err.

309  /// requested by the user/API caller, such as the result of an API function
310  /// requested through the command line, or an explicitly captured
311  /// stdout/stderr stream.
312  DQCS_LOG_NOTE = 4,
313  /// This loglevel is to be used for reporting information NOT specifically
314  /// requested by the user/API caller, such as a plugin starting up or
315  /// shutting down.
316  DQCS_LOG_INFO = 5,
317  /// This loglevel is to be used for reporting debugging information useful
318  /// for debugging the user of the API provided by the logged instance.
319  DQCS_LOG_DEBUG = 6,
320  /// This loglevel is to be used for reporting debugging information useful

322  /// normally only be generated by debug builds, to prevent them from

710 /// Type for a handle.
711 ///
712 /// Handles are like pointers into DQCsim's internal structures: all API calls
713 /// use these to refer to objects. Besides the object, they contain type
714 /// information. This type can be retrieved using `dqcs_handle_type()`.
715 ///
716 /// Handles are always positive integers, counting upwards from 1 upon
717 /// allocation, and they are not reused even after being deleted. Thus, every
718 /// subsequent object allocation returns a handle one greater than the
719 /// previous. Note however that DQCsim may allocate objects as well without
720 /// the user specifically requesting this, so external code should generally
721 /// *not* rely on this behavior unless otherwise noted. The value zero is
722 /// reserved for invalid references or error propagation.
723 ///
724 /// Note that the scope for handles is thread-local. That is, data referenced
725 /// by a handle cannot be shared or moved between threads.
726 ///
727 /// The value zero is reserved for invalid references or error propagation.

1535 /// Constructs a new gate map.
1536 ///>
1537 ///> Returns a handle to a gate map with no mappings attached to it yet. Use
1538 ///> `dqcs_gm_add_*()` to do that. The mappings are queried in the order in
1539 ///> which they are added, so be sure to add more specific gates first. Once
1540 ///> added, use `dqcs_gm_detect()` to detect incoming DQCsim gates, and
1541 ///> `dqcs_gm_construct*()` to (re)construct gates for transmission.
1542 ///>
1543 ///> Gate maps objects retain a cache to speed up detection of similar DQCsim
1544 ///> gates: if a gate is received for the second time, the cache will hit,
1545 ///> avoiding recomputation of the detector functions. What constitutes
1546 ///> "similar gates" is defined by the two booleans passed to this function. If
1547 ///> `strip_qubit_refs` is set, all qubit references associated with the gate
1548 ///> will be invalidated (i.e., set to 0), such that for instance an X gate
1549 ///> applied to qubit 1 will be considered equal to an X gate applied to qubit
1550 ///> 2. If `strip_data` is set, the `ArbData` associated with the incoming
1551 ///> gate is removed.
1552 ///>
1553 ///> Gates are identified through user-defined `void*` keys. To do the above,
1554 ///> however, DQCsim needs to know the following things:
1555 ///>
1556 ///> - how to delete an owned copy of a key if your semantics are that DQCsim
1557 ///> owns it,
1558 ///> - how to compare two keys (equality);
1559 ///> - how to hash a key.
1560 ///>
1561 ///> The deletion function is passed when the key is passed. If the keys are
1562 ///> objects of different classes, this allows different constructors to be
1563 ///> passed here. There can only be a single comparison and hash function for
1564 ///> each gate map, though. They are passed here.
1565 ///>
1566 ///> `key_cmp` represents this comparison function. It takes two `void*` to
1567 ///> keys and must returns whether they are equal or not. If not specified,
1568 ///> the default is to compare the pointers themselves, instead of the values
1569 ///> they refer to. `key_cmp` must be a pure function, i.e., depend only on its
1570 ///> input values.
1571 ///>
1572 ///> `key_hash` represents the hashing function. It takes a `void*` key and
1573 ///> returns a 64-bit hash representative of the key. **For any pair of keys
1574 ///> for which `key_cmp` returns true, the hashes must be equal.** The default
1575 ///> behavior depends on whether `key_cmp` is defined: if it is, all keys will
1576 ///> have the same hash; if it isn't, the pointer is itself hashed. `key_hash`
1577 ///> must be a pure function, i.e., depend only on its input values.
1578 ///>
1579 ///> It is recommended to first preprocess incoming gates with
1580 ///> `dqcs_gate_reduce_control()`. In this case, controlled unitary gate
1581 ///> matrices will be reduced to their non-controlled submatrix, such that the
1582 ///> unitary gate detectors will operate on said submatrix. The predefined
1583 ///> unitary gate detectors are more-or-less based on this assumption (as there
1584 ///> are no predefined controlled matrices).
1585 ///>
1586 ///> Alternatively, you can preprocess with `dqcs_gate_expand_control()`. In
1587 ///> this case, you can use `dqcs_gm_add_fixed_unitary()` to detect the full
1588 ///> matrix in all cases, by specifying the CNOT matrix instead of an X matrix
1589 ///> with one control qubit.
1590 ///>
1591 ///> If you don't preprocess, the upstream plugin determines the
